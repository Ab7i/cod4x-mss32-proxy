# Option B — investigation findings (no code written yet)

## Why Option A failed

mss32.dll proxy uses `SHGetKnownFolderPath(FOLDERID_LocalAppData)`, NOT the
`LOCALAPPDATA` environment variable. Windows ignores env-var overrides for
this API — it reads the user's actual AppData path from registry/SHELL32.

Differential test: hiding our `launcher.dll` in fake-appdata did not break
the launch — game still booted with "Call of Duty 4 X" title. So the real
AppData launcher was being used the whole time.

## Loader chain (verified)

```
iw3mp.exe (PE imports)
  └─ mss32.dll (proxy, in game folder)
       ├─ LoadLibraryA("miles32.dll") [for Miles sound forwarding]
       ├─ SHGetKnownFolderPath(LocalAppData)        ← bypasses env var
       ├─ append L"\\CallofDuty4MW\\bin\\launcher.dll"
       ├─ LoadLibraryW(<path>)
       └─ GetProcAddress("StartLauncher") → call
             └─ launcher.dll!StartLauncher
                  ├─ reads %LOCALAPPDATA% (env var, here it DOES use it)
                  ├─ scans `<LOCALAPPDATA>\CallofDuty4MW\bin\cod4x_*\` with regex `^cod4x_([0-9]+).dll`
                  ├─ LoadLibraryExW(cod4x_021.dll)
                  ├─ GetProcAddress("GetCoD4xVersion") ← entry point check
                  ├─ patches iw3mp's WinMain to redirect to cod4x_021!WinMain
                  └─ returns
```

## Key facts

| Component | Fact |
|---|---|
| launcher.dll | Rust binary (1.1.3) from `callofduty4x/CoD4x-launcher` GitHub repo |

## Proposed Option B1 — custom mss32 proxy (minimal)

### What we write

```
D:\Cod4 Project\mss32-proxy\
  ├── CMakeLists.txt                 (~15 lines)
  ├── main.c                         (~80 lines — DllMain + LoadLibrary chain)
  ├── exports.def                    (341 lines — 340 Miles forwarders + DllMain)
  └── gen-exports.ps1                (~20 lines — regenerates exports.def from miles32.dll)
```

### What our `main.c` does

1. In `DllMain(DLL_PROCESS_ATTACH)`:
   - `GetModuleFileNameW(hSelf, ...)` → find our own path
   - Strip filename → `D:\Cod4 Project\game-test\`
   - Build `D:\Cod4 Project\game-test\fake-appdata` and `SetEnvironmentVariableW(L"LOCALAPPDATA", ...)`
   - Build `D:\Cod4 Project\game-test\fake-appdata\CallofDuty4MW\bin\launcher.dll`
   - `LoadLibraryW` it (absolute path — bypasses SHGetKnownFolderPath)
   - `GetProcAddress("StartLauncher")` → call
   - Return from DllMain
2. launcher.dll then reads `LOCALAPPDATA` (which IS now overridden — launcher uses env var, not Known Folders) and finds our cod4x_021.dll.

### The .def forwarders (no C code per function)

Each line looks like:
```
_AIL_startup@0 = miles32._AIL_startup@0
DLSClose = miles32.DLSClose
... (340 lines total)
```

Windows' loader resolves these at load time — zero runtime cost, zero code.
Auto-generated by `gen-exports.ps1` running `objdump -p miles32.dll`.

## Known risks / things to flag

1. **launcher.dll has GitHub auto-update logic.** It may attempt to verify
   our `cod4x_021.dll` SHA against GitHub release and try to overwrite it
   with the official version. Mitigation options:
   - Test what actually happens (it may handle network failures gracefully)
   - Add a Windows Firewall outbound block for iw3mp.exe in game-test only
   - Patch the launcher (heavy, not recommended)

2. **launcher.dll has integrity checks.** Strings include "CoD4x asset
   integrity verification failed". If our DLL hash differs from the
   manifest, launcher may refuse to load. Need to test.

3. **DllMain restrictions.** Calling LoadLibrary from DllMain is risky.
   The original mss32 proxy does it anyway and works in practice. We'll
   follow the same pattern.
